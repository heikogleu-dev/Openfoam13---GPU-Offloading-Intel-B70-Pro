# Finding 29: ROOT CAUSE — CR 26.18 Aborts on Multi-Process Level-Zero Init (pure L0, no SYCL/Ginkgo/OGL)

## TL;DR

The "CR 26.18 multi-rank race" ([Finding 25](25_cr_26.18_multirank_pthread_race.md))
and the "OGL DEVICE_LOST" ([Finding 27](27_cr2605_ld_switch_workaround.md))
have a single, precise root cause, now isolated with a minimal
pure-Level-Zero MPI reproducer that uses **no SYCL, no Ginkgo, no OGL,
no OpenFOAM**:

> **The Compute Runtime 26.18 Intel-GPU driver
> (`libze_intel_gpu.so.1.15.38308`) calls `abort()` inside `zeInit`
> whenever two or more processes initialise the same B70 GPU. CR 26.05
> does not.**

The earlier "pthread_once race" framing was a red herring — the
`pthread_once` frames in the stack are just the wrapper around
`zeInit`'s one-time init; the actual fault is an `abort()` raised by
the driver itself.

## Minimal reproducer

`findings/code/gpu-diag/diag-mpi-l0.cpp` — ~120 lines, links only
`libze_loader` and MPI. Each rank does:
`zeInit` → `zeDriverGet` → `zeDeviceGet` → `zeContextCreate` →
`zeMemAllocDevice(1 MiB / 100 MiB)`.

```bash
# CR 26.18 (system default): aborts at any np >= 2
mpirun -np 2 ./diag-mpi-l0     # ABORT
mpirun -np 8 ./diag-mpi-l0     # ABORT

# CR 26.05 (user-side LD-switch, findings/27): all ranks pass
LD_LIBRARY_PATH=~/intel-cr-26.05/usr/lib/x86_64-linux-gnu:$LD_LIBRARY_PATH \
mpirun -np 8 ./diag-mpi-l0     # all 8: ALL DONE
```

## The abort

Each rank prints:

```
Abort was called at 15 line in file:
```

The `__FILE__` is empty in the release-stripped 26.18 driver. **The
26.14 driver (Issue #922) printed the full path for the same abort:**

```
Abort was called at 15 line in file:
  ../../neo/shared/source/gmm_helper/resource_info.cpp
```

So the abort is in NEO's **GMM (Graphics Memory Management) helper,
`resource_info.cpp:15`** — the `GmmResourceInfo` creation path, which
queries the kernel xe driver for memory/tiling capabilities during
`zeInit`. (An earlier draft of this finding guessed the IPC socket
server from nearby driver strings — that was wrong; the upstream issue
identifies the exact NEO source line.)

Stack of the aborting rank:

```
abort
libze_intel_gpu.so.1 (+0xfc682)   ← driver calls abort() itself
libze_intel_gpu.so.1 (+0xfc693)
libze_intel_gpu.so.1 (+0x81ca35)
... (driver internal init frames)
libze_intel_gpu.so.1 (+0x1d4a7b)
libze_loader.so.1 (zeInit, via pthread_once)
libur_adapter_level_zero.so.0
```

## This is a known (our own) upstream issue — now sharpened

A web search for this signature returns exactly one public report:
[intel/compute-runtime#922 (GSD-12696)](https://github.com/intel/compute-runtime/issues/922)
— which is our own earlier filing from the CR 26.05 → 26.14 regression
([Finding 13](13_stack_update_zeinit_race.md)). As of this writing it
is still **open, "Needs Feedback", no Intel maintainer response, no fix
version**. The only documented workaround is pinning to CR 26.05.

What this round adds beyond #922:

1. **The regression persists through CR 26.18** (latest as of June 2026;
   no 26.19+ exists). Not fixed in three releases.
2. **A kernel upgrade does NOT fix it.** #922 was on kernel 7.0.0-15;
   we are now on **7.0.0-22** and the abort is identical. This weakens
   the "ABI mismatch with the xe kernel module" hypothesis from #922 —
   a newer xe module did not help.
3. **A pure-Level-Zero minimal reproducer** (`diag-mpi-l0.cpp`, ~120
   lines, no SYCL / Ginkgo / OGL / OpenFOAM). #922's reproducer was at
   the SYCL/MPI application level; this removes every layer above L0,
   making it trivial for Intel to reproduce and bisect.
4. **Staggering ranks does not help** → deterministic, not a timing
   race.
5. **No env-var workaround.** The NEO IPC/GMM debug keys
   (`EnablePidFdOrSocketsForIpc`, `ForceIpcSocketFallback`,
   `EnableIpcSocketFallback`, etc. from
   `shared/source/debug_settings/debug_variables_base.inl`) are not
   honoured as plain environment variables in the release driver — all
   four tested combinations still abort.

## Characterisation

| Configuration | Result |
|---|---|
| CR 26.18, np=1 | ✅ works (single process) |
| CR 26.18, np=2 | ❌ abort |
| CR 26.18, np=8 | ❌ abort |
| CR 26.18, np=8, staggered 200 ms/rank | ❌ abort (timing does NOT help) |
| CR 26.05 (LD-switch), np=8 | ✅ all ranks pass |

**Staggering does not help** — so it is not a timing race but a
deterministic "second process on the device aborts" condition. This
matches an IPC-server bind/connect that fails deterministically when a
second process attaches.

## IPC env-var attempts — did NOT work

Tried setting the driver's IPC-related NEO debug keys directly as
environment variables, with `diag-mpi-l0 -np 8` on CR 26.18:

| ENV variable | Result |
|---|---|
| `EnablePidFdOrSocketsForIpc=1` | ❌ abort |
| `EnablePidFdOrSocketsForIpc=0` | ❌ abort |
| `ForceIpcSocketFallback=0` | ❌ abort |
| `EnableIpcSocketFallback=0` | ❌ abort |

NEO debug keys in a release driver are likely not honoured as plain
environment variables (they need a `NEO_*`-prefixed name or an
`igdrcl.config` file). Not pursued further — the CR 26.05 LD-switch is
the reliable fix.

## The fix that works

[Finding 27](27_cr2605_ld_switch_workaround.md)'s CR 26.05
`LD_LIBRARY_PATH` switch. Confirmed here at the pure-L0 layer:
`diag-mpi-l0 -np 8` passes cleanly with the switch, aborts without it.
The switch swaps only `libze_intel_gpu.so` (the Intel GPU backend);
the rest of the L0 loader stays on the system version.

## Layer attribution — final picture

Combining findings 25-29, the full stack status on BMG-G31:

| Layer | CR 26.18 multi-rank | CR 26.05 (LD-switch) multi-rank |
|---|---|---|
| Pure Level Zero (`diag-mpi-l0`) | ❌ driver abort in zeInit | ✅ pass np=8 |
| Pure SYCL + `ONEAPI_DEVICE_SELECTOR=level_zero:0` (`diag-mpi-sycl`) | ❌ (same driver abort) | ✅ pass np=8 |
| Ginkgo + MPI_Barrier (`diag-mpi-ginkgo`) | ❌ (same driver abort) | ✅ pass np=8, 200 MiB/rank |
| OGL + OpenFOAM, 34M cells | ❌ | ⚠️ BJ(1) ran this morning on a freshly-booted GPU (94 s/step); after a day of crash-cascades the same run hits DEVICE_LOST on the heavier 34M allocation even though the lighter diags still pass — needs a clean boot |

So: **the L0/SYCL/Ginkgo layers are fully fixed by the CR 26.05
LD-switch.** The only remaining wrinkle is that the full 34M-cell OGL
run needs a *freshly booted* GPU — after many crash cascades in a
session the xe driver enters a degraded state (lighter workloads pass,
the big one doesn't) that only a reboot fully clears. This is the
"non-killed execqueue" zombie documented in
[Finding 28](28_gpu_layer_diagnostic_tool_findings.md).

## Upstream-actionable (this is the big one)

This is a clean, minimal, **Intel Compute Runtime** bug report:

- Component: `intel-compute-runtime` / `libze_intel_gpu`
- Version: 26.18.38308.1 (regression vs 26.05.37020.3)
- Repro: `diag-mpi-l0.cpp` + `mpirun -np 2`
- Symptom: `abort()` in `zeInit` IPC-socket-server init with ≥2
  processes per GPU
- Workaround: none found via env vars; downgrade `libze_intel_gpu`
  to 26.05

Multi-process single-GPU is the **normal** mode for MPI CFD (and many
other HPC workloads), so this is a significant regression for any
multi-rank Level Zero use on Battlemage, not just our OGL case.

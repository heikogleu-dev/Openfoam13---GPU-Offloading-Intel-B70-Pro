# Finding 25: Compute Runtime 26.18 Reproduces Multi-Rank `pthread_once`/`zeInit` Race on BMG-G31

## Update — same pattern as Finding 13, second CR major version affected

Compute Runtime version drift: the system was upgraded from
**CR 26.05.37020.3-1** (pinned per [Finding 13](13_stack_update_zeinit_race.md))
to **CR 26.18.38308.1-0** (current `apt list --installed`). The `apt-mark hold`
that previously pinned the stack was apparently released during an unrelated
`apt upgrade`. With this new CR, the multi-rank Ginkgo+OGL pipeline that
previously worked on Ginkgo 1.11 (Finding 19) and on the just-built
Ginkgo 2.0 stack (Finding 24) now hits a `pthread_once`/`zeInit` race during
SYCL platform init.

This is the **second** Compute Runtime major-version bump that broke
multi-rank OGL on BMG-G31 (after CR 26.14 documented in
[Finding 13](13_stack_update_zeinit_race.md)).

## System state at observation

```
apt list --installed:
  intel-opencl-icd            26.18.38308.1-0
  libze-intel-gpu1            26.18.38308.1-0
  libze1                      1.28.2-2
  libze-intel-gpu-raytracing  1.2.3-2

apt-mark showhold: (empty)
```

The hold on the 26.05 packages was no longer in place at the time of this
observation; the user explicitly chose to stay on 26.18 going forward
rather than roll back.

## Reproducer

```
fvSolution: GKOCG + executor sycl + BJ(maxBlockSize=1)
            forceHostBuffer true + ranksPerGPU 8 (or 1)

mpirun -np 8 foamRun -parallel -solver incompressibleFluid
mpirun -np 2 foamRun -parallel -solver incompressibleFluid
```

Both crash identically. Stack tail:


Crash is in the `pthread_once`-guarded init of the Level Zero adapter
(both V1 `libur_adapter_level_zero.so.0` and V2
`libur_adapter_level_zero_v2.so.0` — tested both via
`SYCL_UR_USE_LEVEL_ZERO_V2=0`).

## ENV-var workarounds tested — none cures np=8

Tested with the canonical L0/SYCL workaround env-vars:

| ENV variable | np=2 outcome | np=8 outcome |
|---|---|---|
| `SYCL_UR_USE_LEVEL_ZERO_V2=0` (force V1 adapter) | hangs after `Time = 1s` (timeout) | pthread_once race |
| `SYCL_PI_LEVEL_ZERO_USE_IMMEDIATE_COMMANDLISTS=0` | hangs after `Time = 1s` (timeout) | pthread_once race |
| `ZE_FLAT_DEVICE_HIERARCHY=FLAT` | hangs after `Time = 1s` (timeout) | (not retested — pattern clear) |
| `ZE_FLAT_DEVICE_HIERARCHY=COMPOSITE` | hangs after `Time = 1s` (timeout) | (not retested) |
| All combined | hangs after `Time = 1s` (timeout) | (not retested) |

**Pattern:** With low rank counts the env-vars suppress the race but the run
hangs early during OGL init (potentially because each rank then holds 17M
cells in matrix form rather than 4.25M — the test was a 200 s timeout, may
not be enough for the larger per-rank matrix migration to GPU). With np=8
no env-var prevents the concurrent `zeInit` race.

`np=1` is not an option — OGL requires parallel runs
(`ExecutorHandler.hpp:31: Only parallel runs are supported for OGL`).

## Implications

- Single-rank OGL is structurally not testable
- Multi-rank OGL on BMG-G31 + CR 26.18 cannot complete a solve through
  the existing path (np=8 crashes, low-np hangs)
- The `find_blocks` underflow / ICT / ISAI retests planned for Ginkgo 2.0
  (A1-A3 in the morning plan) are blocked at the OGL multi-rank gate, not
  at the Ginkgo algorithm
- Ginkgo 2.0 algorithm-correctness verification on BMG hardware needs a
  **standalone Ginkgo test program** that bypasses OGL+OpenFOAM and
  decides for itself whether to use MPI or not (see Finding TBD on
  standalone Ginkgo test)

## Isolation Test — CR 26.18 IS the cause (not the Ginkgo 2.0 build)

A natural concern: could the new OGL+Ginkgo 2.0 PR #168 build itself
be triggering the race, rather than CR 26.18 introducing it?

**Sauberer Test:** restore `libOGL.so.1.11.bak.20260522-2242` (the
build from May 22 that worked on CR 26.05) and run the same BJ(1)
np=8 case against the current CR 26.18 stack:

```
ldd libOGL.so | grep ginkgo
  libginkgo.so.1.11.0 → ...lib/libginkgo.so.1.11.0
mpirun -np 8 foamRun -parallel ...
  RC=134
  Ginkgo version: 1.11.0 ( develop)
  pthread_once+0x19 ...
  libze_loader.so.1(zeInit+0x71) ...
```

**Identical crash.** The Ginkgo 1.11 build that was demonstrably stable
4 weeks ago on CR 26.05 now exhibits the same `pthread_once`/`zeInit`
race on CR 26.18. **The blocker is the Compute Runtime upgrade, not
any change in our OGL/Ginkgo build.**

## Decision

The user (Heiko) chose **not** to roll back to CR 26.05. Rationale: CR
26.18 is the current Intel-shipped version; rollback would require
recurring effort against every system update and the Pioneer
documentation is more valuable showing the actual current-stack state
than an artificially pinned environment.

Going forward:
- Multi-rank OGL CFD on BMG-G31 is **paused** until the upstream
  `pthread_once`/`zeInit` race in CR 26.18 (or its multi-rank
  manifestation) is addressed
- Algorithm-level verification on Ginkgo 2.0 will use standalone
  test programs that do not exercise the broken multi-rank init path
- This is now an **upstream blocker**, comparable in scope to the
  triangular-solver gap that
  [Ginkgo PR #2023](https://github.com/ginkgo-project/ginkgo/pull/2023)
  closed — i.e. waiting on Intel Compute Runtime, not on a workaround
  we can apply ourselves
